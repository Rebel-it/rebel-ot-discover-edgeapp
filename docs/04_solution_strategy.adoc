== Solution Strategy

The following table summarises the fundamental decisions that shape the architecture, linked back to the quality goals defined in section 1.

[options="header", cols="2,3,4"]
|===
| Quality Goal | Decision | Justification

| *Maintainability*
| Port-and-adapter decomposition: core app depends on interfaces, not implementations
| `IScraper`, `IApiClient`, `IUAClientFactory`, and `IClientSamplerFactory` decouple the orchestration logic from the protocol connectors. A new industrial protocol (e.g. Modbus, MQTT) is added as a new project implementing the factory interface -- no changes to `Scraper` or `Application` are needed.

| *Maintainability*
| Dependency injection via `Microsoft.Extensions.DependencyInjection`
| All components are registered in `Program.cs` and resolved through the standard .NET DI container. Each connection project exposes an `AddXxx` extension method for clean, encapsulated registration.

| *Security*
| Credentials injected exclusively as environment variables, never stored in source
| All secrets (IXON bearer token, OPC UA password) are supplied at Docker image build time via `--build-arg` and surfaced inside the container as environment variables. They are never written to files, configuration XML, or logs.

| *Security*
| HTTPS enforced for all IXON Cloud API communication
| `BaseAgent` creates `HttpClient` instances targeting `https://portal.ixon.cloud`. There is no HTTP fallback. Bearer tokens are transmitted only over TLS.

| *Portability*
| .NET 10 console application packaged as a Docker container
| A single `linux/arm64/v8` image runs on the IXON SecureEdge Pro. The same solution builds and runs on macOS and Linux developer workstations via `dotnet run`, with no platform-specific code paths.

| *Portability*
| React wizard frontend application served from ASP.NET Core API project
| The React app is built and bundled into static files at Docker build time

| *Performance Efficiency*
| Single-pass, paged API reads with O(1) duplicate checking
| Variables and tags are fetched once per cycle using a page size of 4,000 (the platform maximum). Addresses are indexed in a `HashSet` for constant-time lookup during the browse walk, keeping memory and time proportional to the number of nodes rather than quadratic.
|===

=== Decomposition Approach

The system is divided into six independently deployable projects, each with a single responsibility:

[mermaid]
....
flowchart LR
    subgraph webApp["Rebelit.OT.Discover.EdgeApp.WebApp"]
        frontendNode["Web UI wizard\ndiscovery &amp; configuration"]
    end

    subgraph apiProject["Rebelit.OT.Discover.EdgeApp.API"]
        backendNode["ASP.NET Core API\ncontrollers + scraper"]
    end

    subgraph ixon["Rebelit.OT.Discover.EdgeApp\n.Connections.IXON"]
        apiClient["ApiClient\nHTTPS REST"]
    end

    subgraph opcua["Rebelit.OT.Discover.EdgeApp\n.Connections.OPCUA"]
        uaClient["UAClient\nOPC UA session"]
    end

    subgraph secureedge["Rebelit.OT.Discover.EdgeApp\n.Connections.SecureEdgePro"]
        edgeDeviceApiClient["ApiClient\nHTTPS REST"]
    end

    subgraph installationWizard["Rebelit.OT.InstallationWizard"]
        InstallWizard["executable installer\nfor edge deployment"]
    end

    frontendNode -->|"HTTP REST"| backendNode
    backendNode -->|"IApiClient"| apiClient
    backendNode -->|"IUAClientFactory"| uaClient
    backendNode -->|"IApiClient"| edgeDeviceApiClient
....

The core project depends only on the *interfaces* (`IApiClient`, `IUAClientFactory`, `IClientSamplerFactory`). The two connection projects implement those interfaces in isolation. This means either connector can be replaced or tested independently without recompiling the orchestration layer.
