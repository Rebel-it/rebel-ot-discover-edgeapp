== Deployment View

=== 7.1 Infrastructure Level 1 -- Production Environment

In production the application runs as two Docker containers on the *IXON SecureEdge Pro*, a Linux ARM64 edge gateway installed in or near the OT control cabinet.
The frontend container is the sole entry point for the browser; it proxies all API calls to the backend container over the shared internal Docker network.

==== Motivation

The SecureEdge Pro bridges the isolated OT network and the public internet.
The frontend container makes the web UI reachable on port 3000; the nginx reverse proxy avoids CORS issues by forwarding all `/api/` requests to the backend on the same Docker bridge network.
The backend auto-detects the SecureEdge Pro host address at startup via the container's default gateway, so no address configuration is required.

==== Infrastructure Nodes

[options="header", cols="2,2,4"]
|===
| Node | Technology | Notes

| Developer Workstation
| macOS / Linux / Windows, Docker Desktop with buildx
| Builds both `linux/arm64/v8` images and pushes them to the device registry. No secrets are baked into the images; credentials are supplied per-request by the browser.

| IXON SecureEdge Pro
| Linux ARM64, limited CPU/RAM, local Docker daemon
| Hosts the container registry on HTTP port 5000 and the local device API on port 80. Connected to both the plant OT network (LAN) and the internet (WAN/LTE).

| Frontend Container (`rebel-ot-discover-edgeapp-react`)
| nginx + React SPA (`nginx:alpine` base)
| Exposes port 80 inside the bridge network, mapped to port 3000 on the device. Serves the static SPA and reverse-proxies all `/api/` requests to the backend. `BACKEND_URL` defaults to `http://rebel-ot-discover-edgeapp:8080`.

| Backend Container (`rebel-ot-discover-edgeapp`)
| .NET 10 ASP.NET Core (`mcr.microsoft.com/dotnet/aspnet:10.0` base)
| Listens on port 8080, internal to the bridge network only. `entrypoint.sh` auto-detects `SECUREEDGEPRO_BaseAddress` from the container's default gateway at startup.

| OPC UA Server
| PLC / SCADA / industrial controller
| Accessible only inside the OT network. Communicates over the native OPC UA binary protocol (`opc.tcp://`).

| IXON Cloud API v2
| SaaS platform at `https://portal.ixon.cloud`
| Publicly accessible HTTPS REST API. Authenticated via IXON service account (application ID + access token supplied per-request by the browser).
|===

=== 7.2 Developer Deployment -- `install_discover_edgeapp.sh`

This is the primary deployment scenario: a developer builds and pushes both images from source and creates the containers on the device in a single script run.

[mermaid]
....
sequenceDiagram
    participant Dev as Developer
    participant Script as install_discover_edgeapp.sh
    participant SEP as SecureEdge Pro\n(Docker + local API)
    participant BX as docker buildx

    Dev->>Script: ./install_discover_edgeapp.sh
    Script->>Dev: Prompt: SecureEdge IP address
    Dev-->>Script: e.g. 172.27.21.1

    Script->>SEP: POST /auth/login (admin credentials)
    SEP-->>Script: session cookie

    Script->>BX: docker buildx create --config buildkitd-secure-edge-pro.toml\n(marks device registry as insecure HTTP)

    Script->>BX: docker buildx build backend\n--platform linux/arm64/v8\n--push → {IP}:5000/rebel-ot-discover-edgeapp:latest\n-f docker/Dockerfile
    BX-->>SEP: push linux/arm64/v8 image

    Script->>BX: docker buildx build frontend\n--platform linux/arm64/v8\n--push → {IP}:5000/rebel-ot-discover-edgeapp-react:latest\n-f src/WebApp/docker/DockerFile
    BX-->>SEP: push linux/arm64/v8 image

    Script->>SEP: POST /api/v1/docker/containers (backend)\nnetwork: machine-builder, no external ports
    Script->>SEP: POST /api/v1/docker/containers (frontend)\nnetwork: machine-builder, port 3000→80

    Script->>SEP: POST /api/v1/docker/containers/rebel-ot-discover-edgeapp/start
    Script->>SEP: POST /api/v1/docker/containers/rebel-ot-discover-edgeapp-react/start

    Script->>Dev: "Visit http://{IP}:3000"
....

Key points:

* The build stage uses `--platform=$BUILDPLATFORM` to run the .NET SDK natively on the developer's machine and cross-compiles to `linux-arm64` via `dotnet publish -r linux-arm64 --no-self-contained`. No QEMU emulation is needed.
* No credentials or secrets are baked into the images. The frontend nginx config only stores the backend container hostname (`BACKEND_URL`). All IXON and OPC UA credentials are supplied by the browser at runtime.
* Containers are created and started via the SecureEdge Pro local Docker API using a session cookie obtained at login -- no SSH access to the device is needed.

=== 7.3 End-User Deployment -- `deploy_discover_edgeapp.sh`

For end-user installations, Rebelit ships a release zip containing pre-built Docker image tarballs. The user runs a single script that loads, tags, and pushes the images and then creates and starts the containers -- identical result to the developer scenario but without a build step.

[options="header", cols="2,4"]
|===
| Step | Action

| Prerequisites
| Extract the release zip. Ensure `{device-ip}:5000` is listed as an insecure registry in Docker Desktop settings.

| Load & push images
| `docker load` reads `rebel-ot-discover-edgeapp.tar` and `rebel-ot-discover-edgeapp-react.tar`, tags each image for the device registry, and pushes them to `{device-ip}:5000`.

| Create & start containers
| Identical to the developer scenario: containers are created via the SecureEdge Pro Docker API on the `machine-builder` bridge network, then started.

| Access
| Browser navigates to `http://{device-ip}:3000`.
|===

=== 7.4 Development Environment

Developers can run the backend locally with `dotnet run` from `src/Rebelit.OT.Discover.EdgeApp.API/`. The frontend is served by the Vite dev server (`npm run dev` from `src/Rebelit.OT.Discover.EdgeApp.WebApp/`). The `SECUREEDGEPRO_BaseAddress` environment variable must be set manually to point at the device (or a mock).

There are no environment-specific build flags or configuration files. Switching to production is purely a matter of deploying via the install script.
