[[section-technical-risks]]
== Risks and Technical Debt

The following table lists identified risks and areas of technical debt, ordered by priority. Each item includes a suggested mitigation.

[options="header", cols="1,2,1,1,4,4"]
|===
| ID | Title | Likelihood | Impact | Description | Mitigation

| R-01
| *Credentials stored in browser session storage*
| Low
| Medium
| The IXON access token, application ID, and OPC UA password are no longer baked into the Docker image. They are entered by the user in the wizard and stored in browser `sessionStorage` for the duration of the session. `sessionStorage` is readable by any JavaScript running on the same origin, so a stored XSS vulnerability in the React application could expose these values. Credentials are cleared when the browser tab or window is closed.
| Ensure the React application has no XSS vulnerabilities (avoid `dangerouslySetInnerHTML`, sanitise any user-controlled content rendered to the DOM). The application only serves static assets and does not accept user-generated content, which significantly limits the attack surface.

| R-02
| *OPC UA certificates auto-accepted*
| Low
| High
| `AutoAcceptUntrustedCertificates = true` disables server certificate validation for OPC UA sessions. A machine-in-the-middle attacker on the OT network segment could impersonate the OPC UA server and feed fabricated node data into IXON.
| Acceptable on fully isolated OT networks. If the OT network includes untrusted devices or wireless segments, replace auto-accept with an explicit trusted-certificate store managed by the application. Upgrade `MinimumCertificateKeySize` from 1024 to at least 2048 bits.

| R-03
| *4,000-entry page size is a hard platform limit*
| Medium
| Medium
| IXON Cloud API v2 returns at most 4,000 `Variable` or `Tag` entries per request. If a device has more than 4,000 OPC UA nodes, nodes beyond the first 4,000 will be re-attempted on every cycle without being deduplicated correctly -- the existing address set will be incomplete.
| Add pagination support to `ApiClient.GetDataVariablesAsync` and `ApiClient.GetTagsAsync` using the IXON API cursor or offset mechanism. Track a `next-page` token and accumulate all pages before building the deduplication `HashSet`.

| R-04
| *Single-agent scope per wizard session*
| Low
| Low
| The wizard guides a user through a single agent per session. There is no bulk or multi-agent workflow; a user who needs to configure multiple agents must complete the wizard separately for each one.
| Acceptable for the current use case where configuration of a single agent is the primary workflow. If multi-agent bulk discovery becomes a requirement, extend the wizard with an agent selection step or add a batch API endpoint.

| R-05
| *`HttpClient` instances created per API request*
| Low
| Low
| `BaseAgent.CreateHttpClient()` creates a new `HttpClient` for every GET or POST call. This closes and reopens TCP connections on every request, which consumes extra sockets and DNS lookups and can cause socket exhaustion under high throughput.
| Replace per-request `HttpClient` construction with `IHttpClientFactory` (registered via `AddHttpClient()` in the DI container), which manages a connection pool and respects DNS TTL.

| R-06
| *No integration test coverage for the OPC UA adapter*
| Medium
| Low
| `Connections.OPCUA` has no automated tests. `UAClient` and `ClientSamples` are wrappers around the OPC Foundation SDK and are exercised only in production against a real PLC.
| Add integration tests using the OPC UA reference server (`OPCFoundation.NetStandard.Opc.Ua.Server`) running in-process or in a Docker container with a known address space. Cover at least: successful connection, browse depth, and graceful handling of connection failure.

| R-07
| *Discovery run duration not bounded*
| Low
| Low
| A discovery run over a very large address space (thousands of nodes, slow IXON API responses) can take an arbitrarily long time. There is no timeout, so the user is left waiting at the wizard with no upper bound on when the run will complete or fail.
| Pass a `CancellationTokenSource` with a configurable timeout to `scraper.ExecuteVariableScraperAsync`. If the run does not complete in time, cancel it and return a structured timeout error to the frontend.
|===
