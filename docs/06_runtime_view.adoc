[[section-runtime-view]]
== Runtime View

This section documents three representative runtime scenarios that cover the architecturally relevant behaviour of the system.

=== 6.1 Normal Discovery Cycle

This is the primary scenario: `Application` wakes up, delegates to `Scraper`, which browses the OPC UA server and synchronises new nodes into IXON.

[mermaid]
....
sequenceDiagram
    participant App as Application
    participant Sc as Scraper
    participant UF as UAClientFactory
    participant UA as UAClient
    participant CS as ClientSamples
    participant AC as ApiClient
    participant IXON as IXON Cloud API v2
    participant OPC as OPC UA Server

    loop Every 15 minutes
        App->>Sc: ExecuteAsync(ct)

        alt IXON_DataSourceId env var is set
            Sc->>Sc: Use provided DataSourceId
        else IXON_DataSourceId is absent
            Sc->>AC: GetDevicesAsync(agentId)
            AC->>IXON: GET /api/agents/{id}/devices
            IXON-->>AC: Response<Device[]>
            AC-->>Sc: matched Device

            Sc->>AC: GetDataSourcesAsync(agentId)
            AC->>IXON: GET /api/agents/{id}/data-sources
            IXON-->>AC: Response<DataSource[]>

            alt Existing data source found (device.publicId + slug "opcua" match)
                AC-->>Sc: Existing DataSource
                Sc->>Sc: Reuse existing DataSourceId (no POST)
            else No matching data source exists
                Sc->>AC: PostDataSourceAsync(agentId, newOpcUaDataSource)
                AC->>IXON: POST /api/agents/{id}/data-sources
                IXON-->>AC: Response<DataSource> (with publicId)
                AC-->>Sc: Response<DataSource>
                Sc->>Sc: Store created DataSourceId
            end
        end

        Sc->>UF: Create(address, username, password)
        UF->>UA: new UAClient(config, telemetry)
        UA->>OPC: ConnectAsync (endpoint selection + UserIdentity auth)
        OPC-->>UA: session established
        UA-->>UF: UAClient
        UF-->>Sc: UAClient

        Sc->>CS: CreateAsync()
        CS-->>Sc: ClientSamples

        Sc->>CS: BrowseFullAddressSpaceAsync(client, RootFolder, ct)
        CS->>OPC: Browse (recursive, max depth 128)
        OPC-->>CS: ReferenceDescriptionCollection
        CS-->>Sc: ReferenceDescriptionCollection

        Sc->>AC: GetDataVariablesAsync(agentId)
        AC->>IXON: GET /api/agents/{id}/data-variables?page-size=4000
        IXON-->>AC: Response<Variable[]>
        AC-->>Sc: Response<Variable[]>

        Sc->>AC: GetTagsAsync(agentId)
        AC->>IXON: GET /api/agents/{id}/data-tags?page-size=4000
        IXON-->>AC: Response<Tag[]>
        AC-->>Sc: Response<Tag[]>

        Note over Sc: Build HashSet of existing addresses and tag IDs

        loop For each ReferenceDescription not yet in IXON
            Sc->>AC: PostVariableAsync(agentId, variable)
            AC->>IXON: POST /api/agents/{id}/data-variables
            IXON-->>AC: Response<Variable> (with publicId)
            AC-->>Sc: Response<Variable>

            Sc->>AC: PostTagAsync(agentId, tag)
            AC->>IXON: POST /api/agents/{id}/data-tags
            IXON-->>AC: Response<Tag>
            AC-->>Sc: Response<Tag>
        end

        Sc-->>App: (completed)
        App->>App: Task.Delay(15 min)
    end
....

Key points:

* `Application` does not know about OPC UA or IXON -- it simply calls `IScraper.ExecuteAsync`.
* Before connecting to OPC UA, `Scraper` resolves the data source ID via `DataSourceResolver`: it uses the `IXON_DataSourceId` env var when set; otherwise it fetches existing data sources for the resolved device and reuses a matching one (slug `opcua`); only if no match exists does it POST a new OPC UA data source.
* Variables and tags already present in IXON are skipped via `HashSet.Contains` before any POST is issued.
* The delay occurs *after* the scraping completes (in the `finally` block), so a slow cycle does not cause drift.

=== 6.2 Application Startup

The bootstrap sequence that runs once before the first discovery cycle.

[mermaid]
....
sequenceDiagram
    participant OS as Operating System
    participant Prog as Program (Entry Point)
    participant DI as ServiceProvider
    participant App as Application

    OS->>Prog: dotnet Rebelit.OT.Discover.EdgeApp.dll
    Prog->>Prog: Read environment variables\n(IXON_* and OPCUA_*)
    Note over Prog: Throws ArgumentNullException\nif any required variable is absent

    Prog->>Prog: Configure Serilog console logger
    Prog->>DI: Build ServiceCollection\n(AddOPCUAClient, AddIXONClient,\nAddSingleton<Application>, AddSingleton<IScraper,Scraper>)
    DI-->>Prog: ServiceProvider

    Prog->>DI: GetRequiredService<Application>()
    DI-->>Prog: Application (with all dependencies injected)

    Prog->>App: Run(CancellationToken.None)
    App->>App: Start scheduler loop (section 6.1)
....

The fail-fast behaviour on missing environment variables is intentional: the container will exit immediately with a descriptive message rather than failing silently during the first browse cycle.

=== 6.3 OPC UA Session Loss and Reconnection

If the OPC UA server becomes unreachable mid-cycle, `UAClient` handles reconnection transparently via `SessionReconnectHandler`. If the cycle itself throws (e.g. the OPC UA server is down at the start), `Application` catches, logs, and waits for the next cycle.

[mermaid]
....
sequenceDiagram
    participant App as Application
    participant Sc as Scraper
    participant UA as UAClient
    participant RH as SessionReconnectHandler
    participant OPC as OPC UA Server

    App->>Sc: ExecuteAsync(ct)
    Sc->>UA: ConnectAsync(address)
    UA->>OPC: Connect attempt
    OPC--xUA: Connection refused / timeout

    UA-->>Sc: returns null
    Sc-->>App: returns early (logs error)

    App->>App: catch(Exception ex)\nlogger.LogError(...)
    App->>App: Task.Delay(15 min)
    Note over App: Next cycle begins after 15 minutes.\nNo process restart required.

    rect rgb(220,240,220)
        Note over UA,RH: If connection is lost DURING a session:
        OPC--xUA: KeepAlive failure (after 5 s interval)
        UA->>RH: Trigger reconnect (initial 1 s, backoff to 15 s)
        RH->>OPC: Reconnect attempts
        OPC-->>RH: Session restored
        RH-->>UA: Session updated
    end
....

The `Application`-level exception catch ensures that a full OPC UA outage never kills the process. The next cycle will attempt a fresh connection.

=== 6.4 IXON Variable mapping   
We map the Node variables we get from a OPC UA server to IXON Variables and Tags. Ixon only supports specific variables types here is a overview how we do the mapping:
[cols="3*", options="header"]
|==================================================================
| OPCUA Variable    | IXON Variable type | Width 
| bool            | bool               | null  
| uint8           | int                | 8       
| uint16          | int                | 16      
| uint32          | int                | 32     
| uint64          | int                | 64      
| int8            | int                | 8       
| int16           | int                | 16    
| int32           | int                | 32     
| int64           | int                | 64   
| string          | str             | null     
| float           | float              | 32    
| double          | float              | 64
|==================================================================
