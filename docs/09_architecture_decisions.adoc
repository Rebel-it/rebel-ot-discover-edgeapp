== Architecture Decisions

This section documents the architecturally significant decisions in Architecture Decision Record (ADR) format. Decisions that are already summarised in section 4 are cross-referenced rather than repeated in full.

---

=== ADR-01: .NET 10 ASP.NET Core Web API as the Application Model

*Status:* Accepted +
*Date:* 2025 (updated 2026)

*Context:*
The application must run on the IXON SecureEdge Pro, a Linux ARM64 edge gateway with a standard Docker daemon. The team has existing C# expertise. The introduction of the React wizard (ADR-08) required an HTTP API backend to receive requests from the browser.

*Decision:*
Use .NET 10 as the target framework and model the application as an ASP.NET Core Web API (`dotnet Rebelit.OT.Discover.EdgeApp.API.dll`), exposed on port 8080 inside the Docker bridge network. The frontend nginx container proxies `/api/` requests to it.

*Consequences:*

* (+) Cross-platform: builds and runs identically on Linux ARM64 (production) and macOS/Linux/Windows (development) without conditional code.
* (+) HTTP-native: integrates cleanly with the React wizard; standard controller/service/filter patterns apply.
* (+) Containerises cleanly: the multi-stage Dockerfile produces a minimal `aspnet:10.0`-based runtime image.
* (-) No built-in service lifecycle management (restart policies are delegated to the Docker daemon or SecureEdge container orchestration).

=== ADR-02: Port-and-Adapter Decomposition for Protocol Connectors

*Status:* Accepted +
*Date:* 2025

*Context:*
The application needs to support OPC UA as the first industrial protocol, but future protocols (Modbus, MQTT, PROFINET, etc.) are likely. Embedding protocol-specific code directly in the discovery logic would make adding a new protocol a risky modification.

*Decision:*
Decompose the solution into three projects. The core project (`EdgeApp`) defines interfaces (`IApiClient`, `IUAClientFactory`, `IClientSamplerFactory`) and contains only orchestration logic. Each protocol or cloud adapter is a separate project that implements those interfaces and registers itself via an `IServiceCollection` extension method.

*Consequences:*

* (+) Adding a new industrial protocol requires a new project implementing `IUAClientFactory` and `IClientSamplerFactory` -- no changes to `Application` or `Scraper`.
* (+) Each connector can be tested in isolation with a mock or stub replacing the other.
* (-) Increases the number of projects. For the current single-protocol scope this is more structure than strictly necessary.

=== ADR-03: Credentials Baked into the Docker Image via Build Args

*Status:* Superseded by ADR-08 +
*Date:* 2025 (superseded 2026)

*Context:*
This decision is no longer in effect. With the introduction of the React wizard (ADR-08), all IXON and OPC UA credentials are supplied by the browser at runtime as HTTP request headers. The Docker images contain no secrets and do not need to be rebuilt when credentials change.

The original approach of baking credentials into the image as `ENV` variables via `--build-arg` was replaced because it required a full image rebuild and re-deploy for every credential rotation, and because the wizard made per-request credential supply practical.

=== ADR-04: OPC Foundation .NET Standard SDK for OPC UA

*Status:* Accepted +
*Date:* 2025

*Context:*
OPC UA is a complex binary protocol with a rich type system, security profiles, and session management. Implementing a client from scratch is not feasible.

*Decision:*
Use the `OPCFoundation.NetStandard.Opc.Ua` SDK (v1.5.378), the official open-source client library from the OPC Foundation.

*Rejected alternatives:*

* *Third-party OPC UA libraries (e.g. UA-.NET-Standard forks)*: the OPC Foundation SDK is the reference implementation and is actively maintained.
* *Custom implementation*: unreasonable effort and risk.

*Consequences:*

* (+) Full OPC UA specification compliance, including session management, keep-alive, subscription transfer, and security profiles.
* (+) Actively maintained; security patches are available via NuGet upgrade.
* (-) The SDK is large and brings many features (subscriptions, events, method calls) not used by this application. `ClientSamples.cs` is derived from the SDK's reference sample code and carries more surface area than strictly necessary.
* (-) `AutoAcceptUntrustedCertificates = true` is required for many industrial environments but reduces transport-layer security (see Risk R-02).

=== ADR-05: Fixed 15-Minute Polling Cycle

*Status:* Superseded by ADR-08 +
*Date:* 2025 (superseded 2026)

*Context:*
This decision is no longer in effect. The introduction of the React wizard (ADR-08) replaced the autonomous polling loop with on-demand, user-triggered discovery. The operator explicitly initiates a variable discovery run from the wizard; there is no background scheduler. The `Application` class and its `Task.Delay` loop have been removed.

=== ADR-07: Reuse Existing Data Source Instead of Always Creating One

*Status:* Accepted +
*Date:* 2026-03-31

*Context:*
When `IXON_DataSourceId` is not configured, `DataSourceResolver` previously always called `POST /api/agents/{id}/data-sources`. If the device already had a data source with slug `opcua`, the IXON API returned `400 Bad Request` with `"Already used"` on the `slug` property. This caused the application to crash on every startup after the first successful run.

*Decision:*
Before posting a new data source, call `GetDataSourcesAsync` and search the results for an entry whose `device.publicId` matches the resolved device and whose `slug` equals `"opcua"` (case-insensitive). If such a data source is found, return its `publicId` directly without issuing a POST. Only proceed with the POST when no matching data source exists.

*Rejected alternatives:*

* *Catch the 400 and parse the error body*: brittle -- depends on a specific error message string from a third-party API.
* *Always require `IXON_DataSourceId` to be set by the operator*: shifts operational burden onto the installer and breaks the zero-touch goal.

*Consequences:*

* (+) Idempotent: running the application a second time against the same device no longer fails.
* (+) No new API surface required -- `GetDataSourcesAsync` was already available on `IApiClient` and already requested the necessary fields (`device.publicId`, `slug`).
* (+) One additional GET request per startup cycle when `IXON_DataSourceId` is not set.
* (-) The lookup is linear across all data sources for the agent; acceptable given the expected low cardinality.

=== ADR-06: Auto-Accept Untrusted OPC UA Certificates

*Status:* Accepted (with known trade-off -- see Risk R-02) +
*Date:* 2025

*Context:*
Industrial OPC UA servers in the field commonly use self-signed certificates that are not in any trusted certificate store. Requiring manual certificate trust establishment before first run would block the zero-touch onboarding goal.

*Decision:*
Set `AutoAcceptUntrustedCertificates = true` and `MinimumCertificateKeySize = 1024` in `ServiceCollectionExtensions.AddOPCUAClient`. Certificate validation failures are logged but do not prevent session establishment.

*Consequences:*

* (+) Works out of the box against any OPC UA server without certificate pre-configuration.
* (-) Provides no protection against a man-in-the-middle attack on the OPC UA connection. Acceptable on assumed-isolated OT networks; re-evaluate if the network boundary cannot be guaranteed.

=== ADR-09: IXON Service Account Authentication via Request Headers

*Status:* Accepted +
*Date:* 2026

*Context:*
Earlier iterations required an `IXON_BearerToken` environment variable baked into the Docker image, and then explored generating a token from username/password at startup. Both approaches tied credentials to the container lifecycle. With the React wizard supplying context on every request, a cleaner model became available.

*Decision:*
Authentication is based on an IXON service account configured in IXON Cloud. The wizard user supplies the service account's application ID (`Api-Application`) and access token (`Api-Access-Token`) in the login step. These are stored in browser session storage and attached as HTTP headers on every subsequent API call. The `AuthenticationFilter` validates their presence and writes them into the scoped `IIxonAuthenticationContext` for use by all downstream components.

*Consequences:*

* (+) No credentials stored on the device: the Docker images contain no secrets and never need to be rebuilt for a credential change.
* (+) Per-request isolation: each browser session uses its own service account; there is no shared ambient credential in the container.
* (+) The service account access token is managed entirely outside the application (generated and rotated in IXON Cloud).
* (-) The browser must re-enter credentials after a session storage reset (e.g. closing the tab). Acceptable for the wizard use case.

=== ADR-08: React Webwizard

*Status:* Accepted +
*Date:* 2026

*Context:*
The application currently requires users to set multiple environment variables for configuration, which can be error-prone and may not be user-friendly for all operators. A web-based configuration wizard would provide a more intuitive interface for users to input necessary information and manage settings.

*Decision:*
Implement a React web wizard that runs on the edge router and allows users to configure the application through a graphical interface. The wizard will guide users through the necessary configuration steps, validate inputs, and save settings in a way that the application can consume at runtime.
and give the ability to see Variables and the tags that are being scraped and created. And let the user edit them before we push them to IXON.

*Consequences:*

* (+) Improved user experience: a web wizard provides a more intuitive and accessible way for users to configure the application, reducing the likelihood of misconfiguration.
* (+) Real-time feedback: users can see the variables and tags being scraped and created, allowing them to make informed decisions before pushing data to IXON.
* (-) Additional development effort: implementing a web wizard requires additional time and resources, including frontend development and ensuring secure access to the configuration interface.
* (-) API must be created to let the edgeapp and wizard comunicate with each other. This adds complexity to the application architecture and requires careful design to ensure security and reliability.
